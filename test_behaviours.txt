================================================================================
                        dfbench: Test Behaviour Catalogue
================================================================================

This file lists every behaviour that automated tests should verify to guarantee
correctness and stability of the dfbench code-base.  Tests are grouped by
module / layer.  Within each group the items are roughly ordered from the most
fundamental (pure functions, no side effects) to integration-level (requires
setting up full objects).

Priorities:
  [P0]  Must never break: gate CI.
  [P1]  Important invariants: run in CI; a failure is a bug.
  [P2]  Nice to have / slow tests: nightly or manual.

NOTE: Tests that require Differometor/GPU simulation (problem initialization,
actual loss evaluation) must run via SLURM (srun). Tests that only use mocks or
pure-Python logic can run locally.


================================================================================
TEST ORGANIZATION & MARKERS
================================================================================

Recommended file layout under tests/:

  tests/
    conftest.py                   # shared fixtures (mock problem, seeded obj)
    test_api_imports.py           # Section 12: import smoke tests
    test_utils.py                 # Section 1: t2j/j2t/inverse_sigmoid
    test_config.py                # Section 2: create_parser
    test_problem_protocol.py      # Section 3: abstract base
    test_problems_smoke.py        # Section 4: init/loss/grad (slow marker)
    test_problem_bounds_overrides.py  # Sections 4.10-4.11, 4.15, 4.20-4.23
    test_objective_randomness.py  # Section 5 seed & sampling items
    test_objective_invariants.py  # Section 5 all other items
    test_algorithm_protocol.py    # Section 6
    test_algorithms_unit.py       # Section 7 cheap unit tests (mock problem)
    test_metrics.py               # Section 8: full metrics coverage
    test_benchmark_smoke.py       # Section 9: orchestration with mock
    test_display.py               # Section 10
    slow/
      test_problems_full.py       # Section 4 full physics (needs srun)
      test_algorithms_integration.py  # Section 7 + 13 (needs srun)
      test_benchmark_full.py      # Section 13 (needs srun)

Pytest markers (define in pyproject.toml / pytest.ini):
  @pytest.mark.slow   : anything that calls Differometor simulation or runs
                        a real optimizer for >5 s; requires srun on the cluster.
  @pytest.mark.gpu    : tests that assert GPU-specific behaviour or require
                        a CUDA device; must not block CPU-only CI.

Default push suite: everything NOT marked slow or gpu.
Nightly / manual: --run-slow and/or --run-gpu flags enable those markers.

--- Seed & random sampling ---

  RANDOMNESS TESTING PRACTICES (how to write these tests correctly):
  - Always call obj.set_seed(<fixed_int>) at the start of every test that
    involves random sampling; never rely on module-level state.
  - Never use np.random or Python random directly inside tests; only use
    the Objective's set_seed + random_params_* methods so that the seeding
    path being tested matches production code.
  - For property-based tests (bounds, non-NaN), sample n=100 points with a
    fixed seed and assert the property holds for all of them.
  - For reproducibility tests (5.10), compare two independent Objective
    instances both seeded identically; do not reuse the same instance, as
    that would only verify the key-splitting mechanism, not the seed setter.
  - Tests that rely on two consecutive samples being *different* (5.11)
    should assert they are not all-equal, not that they differ element-wise
    (occasional collisions are astronomically unlikely but not impossible).
  - Avoid brittle value snapshots across CPU and GPU backends: JAX float32
    arithmetic is not guaranteed bit-identical between devices.  Seed-based
    reproducibility tests (5.10) must be run on the same device type.
    Use allclose with a generous tolerance (rtol=1e-5) rather than ==.
